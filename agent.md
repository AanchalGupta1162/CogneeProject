# Smart Ticket AI - Bug Context and Handling (Agent Knowledge)

This document serves as a historical record of all major bugs encountered during the development of Smart Ticket AI, including their root causes, context, and the exact solutions implemented. It is designed to provide future LLM agents with the necessary context to avoid repeating these mistakes.

## 1. GitHub Commits Missing from Graph (Sparse Graph)
- **Context**: The `trigger_historical_sync` task was initially passing a raw GitHub URL string to Cognee's `add` endpoint, assuming Cognee would natively clone the repository and extract commits. However, Cognee treated the URL as a simple text string, resulting in an extremely sparse knowledge graph containing only URL nodes and no code or commit history.
- **Handling**: We reverted the pipeline to use our custom `github_client` to manually fetch the last N commits. We formatted each commit (including its author, message, date, SHA, and the full `git diff` patch) into a rich Markdown document. We then injected this Markdown into Cognee using `memory_service.add_repository_data()`. This allowed Cognee's LLM to chunk the markdown properly and build a rich semantic graph of commits, files, and authors.

## 2. Cognee `add` Endpoint Returning `400 Bad Request`
- **Context**: When attempting to ingest data into Cognee, `httpx` was sending the payload as a standard form-encoded body (`application/x-www-form-urlencoded`). However, the Cognee `/api/v1/add` endpoint strictly requires a `multipart/form-data` payload.
- **Handling**: We updated the `httpx` post request in `packages/infrastructure/cognee_client.py` to use the `files` parameter: `files={"data": (None, data.encode('utf-8'), "text/plain")}`. This forced `httpx` to encode the entire request correctly as `multipart/form-data`.

## 3. `httpx.ReadTimeout` During `cognify`
- **Context**: The `cognify` endpoint takes significant time to build the knowledge graph from large markdown chunks. The default `httpx` client timeout is 5 seconds, causing the API call to crash with a `ReadTimeout` before the Cognee worker could finish processing.
- **Handling**: We explicitly increased the timeout for the `cognify_repository` HTTP client to 300 seconds (`httpx.AsyncClient(timeout=300.0)`) to give the Cognee backend ample time to process the data and return the completion response.

## 4. `asyncio.run() cannot be called from a running event loop`
- **Context**: To avoid hitting Google Gemini LLM rate limits during ticket assignment, a custom FastAPI background task (`process_tickets_rate_limited`) was created. It awaited `trigger_assignment_agent(ticket.id)`. However, `trigger_assignment_agent` was designed as a Celery `@shared_task` and internally called `asyncio.run()` to execute the `AssignerAgent`. Because FastAPI already runs on an asynchronous event loop, calling `asyncio.run()` threw a `RuntimeError`.
- **Handling**: We imported `run_in_threadpool` from `starlette.concurrency` and executed `trigger_assignment_agent` inside a separate background threadpool (`await run_in_threadpool(trigger_assignment_agent, ticket.id)`). This safely isolated the synchronous `asyncio.run()` call from FastAPI's main event loop.

## 5. Cognee `search` Endpoint Returning `404 Not Found` (DatasetNotFoundError)
- **Context**: The `AssignerAgent` was receiving `ticket.repository_id` from the SQL database and passing it to the `search_memory` tool as the `dataset_name`. Because `ticket.repository_id` is a UUID (e.g., `448d5572-...`), Cognee attempted to search a dataset with that UUID name. However, the dataset was initially ingested using the repository's GitHub full name (e.g., `Harshavardhan-28/cognee_test`). Cognee responds with a 404 when a requested dataset name does not exist.
- **Handling**: We updated `trigger_assignment_agent` in `packages/agents/orchestrator.py` to query the `Repository` SQL table and fetch the actual repository name (`repo.name`). This GitHub name is now passed to the `AssignerAgent` as the `repo_id`, ensuring the `search_memory` tool targets the correct Cognee dataset.

## 6. `AttributeError: 'str' object has no attribute 'model_dump'`
- **Context**: After fixing the 404 error, the Cognee `search` endpoint successfully returned results. The `search_memory` tool in `packages/agents/tools.py` attempted to iterate over these results and call `.model_dump()`, assuming the results were Pydantic models. However, because the results were generated by `httpx.post(...).json()`, they were already raw Python dictionaries/strings.
- **Handling**: We simplified the `search_memory` tool to directly return `json.dumps(results)`, bypassing the unnecessary `.model_dump()` call.
